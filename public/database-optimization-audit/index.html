Audit shape

Expected deliverables and timing

A useful audit should be evidence-first, reviewable, and limited to recommendations until a human owner approves production changes.

1

Evidence inventory

Queries, wait symptoms, index usage, table growth, configuration notes, capacity pressure, and incident context.

2

Prioritized findings

A short list of likely bottlenecks ranked by impact, confidence, reversibility, and production risk.

3

Safe next steps

Recommended experiments, indexes, query rewrites, maintenance checks, or configuration review points with approval gates.

4

Timing expectation

Expect scoping first. PostgreSQL teams can move into the specialist report path; other engines should treat this as preparation guidance.

Fit check

Who this is for

Good fit

Teams with recurring slow queries, unclear index value, table growth concerns, capacity pressure, or production incidents that need a reviewable database optimization plan.

Not a fit yet

Teams looking for emergency production access, automatic fixes, guaranteed cost reduction, formal support for every database engine, or a paid SaaS workflow on this page.

Safety boundary

The audit should not require production write access. Recommendations should stay read-only and evidence-based until your team reviews, tests, approves, and schedules any production SQL or configuration change.

Preparation

What to have ready

You do not need perfect telemetry, but the request is stronger when the performance complaint is tied to evidence and business impact.

A

Database context

Engine, version, hosting model, workload shape, scale, and the user-visible symptom.

B

Evidence sample

Slow query examples, execution plans, index lists, table sizes, wait notes, or database statistics you can share safely.

C

Change limits

Maintenance windows, approval rules, rollback expectations, data sensitivity, and changes your team will not accept.

Prepare before requesting scope

Use the database optimization checklist when the problem is still broad. Review how the audit works and the read-only audit boundary if access, customer data, or production safety is the main concern.

Move from broad audit to the right evidence path

Start with the database optimization model. Use the broader database performance audit when symptoms are unclear, SQL performance tuning when specific statements drive the risk, and index optimization when access paths need review.
