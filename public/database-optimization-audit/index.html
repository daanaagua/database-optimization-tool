Audit intent

A database optimization audit should explain what is slow and what evidence supports the next step.

Use the audit to turn a vague performance complaint into a reviewable database optimization work plan backed by query, index, table, and capacity evidence.

1

Collect evidence

Start from database statistics, query patterns, table size, and index usage.

2

Prioritize risk

Separate high-impact bottlenecks from cleanup tasks that can wait.

3

Explain tradeoffs

Document why an index, query rewrite, maintenance change, or plan upgrade may help.

4

Require review

Keep human approval in front of production SQL and configuration changes.

Move from broad audit to the right evidence path

Start with the database optimization model. Use the broader database performance audit when symptoms are unclear, SQL performance tuning when specific statements drive the risk, and index optimization when access paths need review. Then use the PostgreSQL path when the database is PostgreSQL.
