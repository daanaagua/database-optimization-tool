Sample report

What you will receive.

The first response is not an automated diagnosis. It is a human-reviewed scope checkpoint that tells you whether the PostgreSQL audit path fits and what evidence should come next.

1

Scope confirmation

A short note confirming whether the issue looks suitable for PostgreSQL audit review, which workload area matters most, and which evidence is worth collecting first.

2

Evidence checklist

A practical list of read-only inputs to prepare, such as PostgreSQL version, schema shape, slow query examples, pg_stat_statements availability, table size signals, and index context.

3

Audit outline

Expected review areas can include slow query pressure, index candidates, unused index signals, table health, bloat risk, autovacuum drift, temp writes, and role review.

4

Next-step recommendation

You receive a human-reviewed recommendation for whether to continue with a fuller PostgreSQL audit, prepare more evidence, or use a narrower slow-query review.

What to prepare before requesting scope.

You do not need to grant production control from this page. A useful request gives enough context for a PostgreSQL specialist to decide which read-only evidence matters. If the broader audit boundary is still unclear, review how the audit works and the audit FAQ first.

Workload and symptom

Prepare the PostgreSQL version, hosting environment, workload type, recent traffic changes, slow endpoints, timeout patterns, lock waits, or cost pressure.

Query and schema context

Bring representative slow queries, EXPLAIN output if available, major table sizes, critical indexes, and whether pg_stat_statements is enabled.

Review slow-query path

Safety constraints

State whether production access is restricted, whether only snapshots are allowed, and what approval process is required before any recommendation can become a change.

What we will not do.

The PostgreSQL path stays evidence-first. Recommendations are reviewed by a human and remain separate from production changes until your team approves them.

A

No production writes

This route does not create indexes, run migrations, rewrite data, change configuration, or execute automated tuning against production.

B

No broad credential request

Use least-privilege, temporary, read-only access where possible. Avoid superuser credentials and permanent access unless a safer path is impossible.

C

No instant automated report claim

The first step is preliminary reply and scope confirmation. A fuller audit depends on the evidence available and the specialist workflow outside this category page.

D

No full SaaS workflow here

This site does not host billing, accounts, uploads, transactions, or managed report delivery. Those workflows belong on the PostgreSQL specialist service.
